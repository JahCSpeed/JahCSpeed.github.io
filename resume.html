Load Balancing Server

Lead Developer
February 2023

Description:

A set of programs that mimic an implementation of load balancing across DNS servers. In its simplest form, a “load balancer” simply picks a single DNS server to respond to any given request, spreading requests as evenly across servers as possible.

Features:

  • TS server looks up a domain name queried by the LS and return an entry with an IP address,
  • Each TS maintains one connection: with the LS
  • Each TS server maintains a DNS table consisting of three fields:
    • Domain name
    • IP address
    • Resource record type
  • For each query received from the LS, each TS server does a lookup of the domain name in its DNS table, and if there is a match, sends a DNS response
  • The LS server receives queries from the client and forwards them directly to both TS1 and TS2. If either one responds, LS will relay the response directly to the client, else it will send an error message.
  • The client sends queries to the LS. The client also directly prints the output it receives from the LS.

Shell

Designer/Lead Developer
October 2021

Description:

A command-line interface that allows users to interact with the operating system. It acts as an intermediary between the user and the underlying operating system, allowing users to execute commands, run programs, and manage files and directories.

Features:

  • Execute other programs, possibly with command-line options, whether they’re specified with an absolute path or relative path
  • If a command is given with no absolute/relative path (e.g., “ls”), and it’s not a built-in command the shell will look in /usr/bin and /bin and execute the first such program found
  • If the command line ends in an ampersand, the shell will run the program in the background. Otherwise, it will run it in the foreground and wait for the executed program to complete before reading another line of input.
  • Ctrl-c sends SIGINT to the foreground job and any of its child processes
  • Ctrl-z should send SIGTSTP to the foreground job and any of its child processes
  • Recognize these built-in commands:
    • bg <jobID>
    • cd [path]
    • exit
    • fg <jobID>
    • jobs
    • kill <jobID>

Cache Simulator

Designer/Lead-Developer
December 2020

Description:

A L1 and L2 cache simulator. The input to the cache simulator is a memory access trace, and uses these addresses to determine if the access is a hit or a miss, and the actions to perform in each case.

Features:

  • Level One Cache:
    • Implements a level one cache simulator
    • Performs both FIFO & LRU replacement policies

    • Simulates a write through cache.
    • Produce a text document with amount of memread,memwrite,cachehit and cachemiss in the simulaton
  • Level Two Cache:
    • Implements a level two cache simulator
    • Performs both FIFO & LRU replacement policies
    • The cache size, associativity, the replacement policy, and the block size are input parameters. Cache size and block size are specified in bytes.
